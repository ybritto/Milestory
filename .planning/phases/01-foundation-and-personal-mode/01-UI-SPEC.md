### Source Notes

- `01-CONTEXT.md` D-05 through D-08 define a small polished Milestory-branded home/status experience instead of starter content.
- No `components.json`, Tailwind config, or reusable frontend UI layer exists in the repo, so the contract stays manual for Phase 1.
- `PROJECT.md` keeps UI quality high but leaves library choice open. Phase 1 should not add a second system before real feature flows exist.

---

## Phase Surface Contract

- Ship a single primary route at `/` for the Milestory home screen.
- The screen must feel like a real product landing surface, not an admin shell and not an Angular starter page.
- The screen must include exactly three visual zones:
  1. Brand intro: Milestory name, one-sentence product promise, one primary action.
  2. Foundation status card: backend connectivity/runtime summary from the narrow Phase 1 API surface.
  3. Build notes strip: 2-3 short bullets explaining what exists now and what arrives in later phases.
- Do not render fake goal charts, fake user profiles, auth prompts, or placeholder CRUD tables.
- Personal mode must stay invisible. Copy must speak as if the app simply belongs to the user already.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge spacing |
| sm | 8px | Compact element spacing, list gaps |
| md | 16px | Default element spacing, card internals |
| lg | 24px | Section padding, form/control groups |
| xl | 32px | Layout gaps between major blocks |
| 2xl | 48px | Major section breaks inside the page |
| 3xl | 64px | Page-level top/bottom breathing room |

Exceptions: `44px` minimum touch target for all clickable controls; `72px` top padding allowed on large screens for the hero section.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 24px | 600 | 1.2 |
| Display | 40px | 600 | 1.1 |

### Typography Rules

- Use exactly two weights in Phase 1: `400` and `600`.
- Display text uses `Fraunces`; body, labels, metadata, and buttons use `Manrope`.
- Limit display type to the hero only.
- Status metadata and support copy stay at body or label size. Do not introduce a fifth text size.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F6F1E7` | App background, large surfaces, hero canvas |
| Secondary (30%) | `#E6DDCF` | Cards, status panel, callout strip, subtle dividers |
| Accent (10%) | `#1F6B5C` | Primary CTA, status-ready badge, focused links, active emphasis line |
| Destructive | `#B64735` | Error badge, failure text, destructive actions only |

Accent reserved for: primary CTA background, ready/healthy status pill, keyboard focus ring, brand wordmark highlight, and one supporting divider or underline. Never use accent as the default color for all text, all icons, or every interactive element.

### Color Rules

- Keep the screen warm and calm rather than neon or dashboard-dark.
- Primary body text should render in a near-ink neutral such as `#1F1A17`.
- Secondary/support text should render in a muted neutral such as `#5F554C`.
- All text and controls must meet WCAG AA contrast against their surfaces.

---

## Interaction Contract

- Load the home screen with a server-status request immediately on first paint.
- Show one of four explicit states in the status card:
  - `loading`: skeleton or quiet placeholder rows, never a spinner-only screen
  - `ready`: success badge, concise summary sentence, and 2-4 key facts
  - `empty`: structured fallback when the endpoint returns no status details yet
  - `error`: failure message with one retry action
- Primary CTA label is `Refresh status`.
- Secondary action, if present, is a quiet text link to API documentation or technical details. Do not add more than one secondary action.
- The page should scroll naturally on mobile, but on desktop the hero and status card should fit above the fold when possible.
- Keyboard focus order must be linear: heading -> primary CTA -> status card content -> secondary link.
- Motion should be restrained: one fade/slide-in on initial load and a subtle status-card transition when data changes. No looping animations.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Refresh status |
| Empty state heading | Foundation is in place |
| Empty state body | Milestory is running, but there are no extra system details to show yet. Continue with setup and check back after the next backend update. |
| Error state | Milestory could not load the current foundation status. Confirm the backend and database are running, then try again. |
| Destructive confirmation | none in Phase 1 |

### Tone Rules

- Product tone should be calm, grounded, and quietly motivating.
- Do not mention authentication, accounts, sign-in, bearer tokens, or personal mode in user-facing copy.
- Do not oversell future features. Acknowledge that goal planning and dashboards arrive in later phases.
- Prefer concrete product language such as `foundation`, `status`, `ready`, and `next phase` over scaffold or template vocabulary.

## Implementation Notes

- Remove all Angular starter branding, gradients, and placeholder SVG content.
- Move visual tokens into global styles or a dedicated theme partial instead of leaving large inline `<style>` blocks in templates.
- Use semantic landmarks: `main`, `section`, `header`, and a list structure for system facts.
- Keep the Phase 1 page read-only. No editable controls beyond retry/refresh navigation.
- If backend status details are still minimal, prefer a compact facts list such as API availability, database migration state, and app mode.
